

tap tradingg project backend.

Create a node js backend to manage the following.
Will be using MongoDB as database.

About the project: Tap trading game, where users can bets on the future out-come of the tokens to win.
User will get a contineous strem of price, on the x axis it'a a time where each tile is 10 secs apart and on the Y axis it's a price,
Within a span of 10 secs, user is able to place the trade, and based on the out-come, like user tap on the tile, we record the tile and once the round is resolve users will either win or lose  

For the gride, the X axis will be the same 10sec. for Y axis should vairy.
E.g. of the token is at price 1000$ then the spread might be +-10$ such that each tile will contain the spread of 10$. 
But this spread will be chang realt-time based on the token, volatility.

Also the winning amount also varies on the user time of bet, actual price at the bet, the differen and all the factors.
For all maths and calculations refer to @calculations.txt for explainign all the formulas and calculations.

Base line: 
- will have taking the 10 sec duration for each session.
- User can bet on the next session, and the current session gets pause for the resolution
- Will be using Binance websockt for the contineous feed.
- Will have to create the Rest endpoint like deposite, withdraw, current balance, trade, etc..
- A websockt connection for the contineous price

```
This table lets you reconstruct any proof path. Given a user and a checkpoint, you query their leaf node, then walk up by finding siblings at each level until you reach the root.

**How it all connects**
```
User deposits on-chain
  → deposits row created
  → users.balance updated
  → receipts row created

User places bet
  → bets row created
  → users.balance decremented
  → users.locked_balance incremented
  → users.nonce incremented
  → receipts row created

Round settles
  → rounds.settlement_price filled
  → rounds.status = settled
  → bets.status = won/lost
  → users.balance updated
  → users.locked_balance decremented
  → receipts row created per user

Every 2 hours
  → Read snapshot of users, rounds, prices
  → Build merkle tree
  → Insert all nodes into merkle_nodes
  → Insert checkpoints row
  → Post root to Sui
  → Update checkpoints.sui_tx_hash

User withdraws
  → Query merkle_nodes for their proof path
  → Create withdrawals row
  → User submits proof to Sui contract
  → Update withdrawals.status on confirmation
